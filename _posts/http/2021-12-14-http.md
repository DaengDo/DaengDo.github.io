> http 학습을 위한 사전 지식

http는 인터넷 통신의 기반 위에서 작동하므로 설명을 위해 기본적인 네트워크 지식이 필요하다.

로컬 환경이라면 서버와 클라이언트를 케이블로 연결해서 간단히 서로 요청과 응답을 주고받을 수 있지만,<BR>인터넷 환경은 대단히 복잡하다.<BR>최선의 방법을 위해 ip 프로토콜이 필요하다.

### 1. 인터넷(ip) 프로토콜

- IP 프로토콜
  - 지정한 IP 주소(IP Adress)에 데이터 전달
  - 패킷이라는 통신 단위로 데이터 전달
    - 패킷은 출발지IP, 목적지IP가 적혀있어서 그 정보를 바탕으로 노드가 목적지까지 전달하게 됨
- IP 프로토콜의 한계
  - 비연결성
    - 패킷을 받을 대상이 없거나 서비스 불능 상태여도 패킷 전송
  - 비신뢰성 
    - 중간에 패킷이 사라지거나, 패킷이 순서대로 안가면?
  - 프로그램 구분
    - 같은 IP를 사용하는 서버에서 통신하는 애플리케이션이 둘 이상이면?

### 2. TCP (+ UDP) 프로토콜

- 인터넷 프로토콜 스택의 4계층
  - <img src="https://upload.wikimedia.org/wikipedia/commons/thumb/e/e5/TCP-IP_Model_-_en.png/640px-TCP-IP_Model_-_en.png" width="50%">
  - TCP는 출발지 PORT, 목적지 PORT, 전송 제어, 순서, 검증정보 등등을 넣어준다.
- TCP 특징
  - 1) 연결지향 (연결 여부 먼저 확인) : `3 way handshake`
    - `SYN`(접속 요청), `SYN + ACK`(요청 수락), `ACK` 으로 메세지를 서로 주고받아서 클라이언트와 서버가 연결되있음을 검증함
    - 3 way handshake를 한 후에 데이터를 전송한다. 요새는 최적화가 되어서 ACK를 할때 데이터도 같이 전송함
  - 2) 데이터 전달 보증 (패킷 누락시 클라이언트가 알 수 있음)
    - 데이터를 전송하고 데이터를 잘 받았다고 응답을 줌
  - 3) 순서 보장
    - 패킷의 순서를 보장해줌
  - 위의 3가지 이유로 신뢰할 수 있는 프로토콜로 간주, 대부분의 애플리케이션에서 TCP를 사용
- UDP 특징(하얀 도화지에 비유)
  - IP와 거의 같음, 포트와 체크섬이 추가됨
  - 애플리케이션에서 추가 작업 필요


### 3. PORT

한번에 둘 이상 연결해야 하면?

- 같은 IP 내에서의 프로세스 구분
  - 애플리케이션들을 구분하기 위한 것
- 0 ~ 65535 할당 가능
  - 0 ~ 1023 : 잘 알려진 포트, 사용하지 않는 것이 좋음
