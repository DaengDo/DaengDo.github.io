# 1. http 메시지로 전송하는 것
- HTML, TEXT
- 이미지, 음성, 영상 파일
- JSON, XML (API)
- 거의 모든 형태의 데이터 전송 가능
- 서버간 데이터 송수신도 대부분 HTTP 사용

## 기반 프로토콜
- <b>TCP</b>: HTTP/1.1, HTTP/2
- <b>UDP</b>: HTTP/3
- 현재 HTTP/1.1 주로 사용

## HTTP 특징
- 클라이언트 서버 구조
- 무상태 프로토콜(stateless), 비연결성
- HTTP 메시지
- 단순함, 확장  가능

# 2. 클라이언트 서버 구조
- Request, Response 구조
- 클라이언트는 서버에 요청을 보내고, 응답을 대기
- 서버가 요청에 대한 결과를 만들어 응답

# 3. 무상태 프로토콜(Stateless)
점원 예시가 떠오르는가?
{: .small}

- 서버가 클라이언트의 상태를 보존하지 않음
- 장점: 서버 확장성 높음(스케일 아웃)
  - 상태 유지
    - 중간에 context가 바뀌면 안됨
  - 무상태
    - 응답 서버를 쉽게 바꿀 수 있음 <BR>-> 무한한 서버 증설 가능 - 수평 확장 유리(스케일 아웃)
- 단점
  - 클라이언트가 많은 데이터를 전송해야함
  - 모든 것을 무상태로 설계 할 수 있는 경우도 있고 없는 경우도 있다.
    - 무상태 예시) 로그인이 필요 없는 단순 소개 화면
    - 상태 유지 예시) 로그인
    - 로그인한 사용자의 경우 로그인 했다는 상태를 서버에 유지
    - 일반적으로 브라우저 쿠키와 서버 세션등을 사용해 상태 유지
  - 따라서 상태 유지는 최소한만 사용

# 4. 비 연결성(Connectionless)
- HTTP는 기본이 연결을 유지하지 않는 모델
- 일반적으로 초 단위 이하의 빠른 속도 응답
- 서버 자원을 효율적으로 사용할 수 있음
- 한계
  - TCP/IP 연결을 매번 새로 맺어야 함 - 3 way handshake 시간 추가
  - HTML, CSS, JS, 추가 이미지 등 수 많은 자원이 함께 다운로드
- 극복
  - 현재 HTTP 지속 연결(Persistence Connections)로 문제 해결
  - HTTP/2, HTTP/3 에서 더 많은 최적화가 이뤄짐

# 5. HTTP 메시지
- HTTP 메시지 구조
  - `start-line` 시작라인
    - request-line (요청)
      - method SP
        - 종류: GET, POST, PUT, DELETE
        - 서버가 수행해야 할 동작을 지정
      - request-target SP
        - `absolute-path[?query]` (절대경로[?쿼리])
        - 절대경로="/" 로 시작하는 경로
      - HTTP-version CRLF
    - status-line (응답)
      - HTTP-version SP
      - status-code SP
      - reason-phrase CRLF(이유 문구: 짧은 코드 설명 글)
  - `header` 헤더
    - header-field = field-name "`:`" OWS field-value OWS<BR>(OWS:띄어쓰기 허용)
      - 예시)<BR>Host: www.google.com<BR>Content-Length: 3424
    - HTTP 전송에 필요한 모든 부가정보
  - `empty-line` 공백라인 (CRLF)
  - `message-body`
    - 실제 전송할 데이터
    - HTML 문서, 이미지, 영상, JSON 등등 byte로 표현할 수 있는 모든 데이터 전송 가능
